# i18n guide (kk/en/ru)

## Frontend
- Stack: `next-intl` with locales `en`, `ru`, `kk`.
- Locale routing config: `frontend/src/i18n/routing.ts`.
- Request config/messages loader: `frontend/src/i18n/request.ts`.
- Middleware for locale detection and cookie handling: `frontend/middleware.ts`.
- Root provider: `frontend/src/app/layout.tsx`.
- Translation catalogs: `frontend/messages/en.json`, `frontend/messages/ru.json`, `frontend/messages/kk.json`.
- Language switcher: `frontend/src/components/LanguageSwitcher.tsx` (sets `NEXT_LOCALE` cookie and `atlas_locale` in localStorage).

## Backend
- i18n core module: `backend/app/i18n/core.py`.
- Supported locales: `en`, `ru`, `kk`, with fallback to `en`.
- Locale resolution priority:
  1. `X-Locale` header
  2. `Accept-Language` header
  3. default `en`
- Global localization is wired in `backend/app/main.py`:
  - request locale middleware (`request.state.locale`)
  - localized `HTTPException` handler
  - localized `RequestValidationError` handler
  - `Content-Language` response header

## How to add new text
- Frontend: add a key to all three files under `frontend/messages/*.json`, then use `useTranslations(...)` in the component.
- Backend error: raise `HTTPException(detail={"key": "errors.someKey"})`, then add `errors.someKey` into backend i18n catalogs.
- Backend validation issue: use stable issue codes and map them in `localize_issue(...)`.

## Notes
- Frontend API sends locale headers via `X-Locale` and `Accept-Language`.
- Keep key names stable (`nav.*`, `pages.*`, `crud.*`, `schedule.*`, `errors.*`, `validation.*`) to avoid breaking translations.
